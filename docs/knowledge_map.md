## ナレッジマップ

本ドキュメントは「ソフトウェアテスト哲学」シリーズ全6文書の構造を俯瞰するための索引である。
個別の技術解説は含まない。各文書の位置づけと依存関係を把握し、学習順序の判断に使う。

---

## 依存関係ツリー

```
  #1 ソフトウェアテストの目的論 [Level 1 / Strategy_Design]
   │
   ├── → #2 境界値分析の思想 [Level 1 / Technology]
   │
   ├── → #3 テストピラミッドの設計判断 [Level 1 / Strategy_Design]
   │        │
   │        ├── → #4 シフトレフトの本質 [Level 2 / Strategy_Design]
   │        │
   │        └── → #5 テスト自動化の判断基準 [Level 2 / Technology]
   │
   └── → #6 品質はテストで作れない [Level 3 / Strategy_Design]
              requires: #1, #4, #5
```

矢印（→）は enables の方向を示す。
#6 は3つの文書（#1, #4, #5）を前提知識として統合する結論ノードである。

## 学習パスの例

**基礎から順に**: #1 → #3 → #4 → #5 → #6（王道ルート）

**テスト技法に興味がある場合**: #1 → #2（境界値分析で完結する独立パス）

**品質戦略を急ぎたい場合**: #1 → #3 → #6（最短で結論へ到達、ただし#4・#5の前提知識が不足する）

## 構造の特徴

- **ルートノード**: #1（唯一の出発点。全文書がここに依存する）
- **シンクノード**: #6（全パスの終着点。3つの前提知識を統合する）
- **独立末端**: #2（#1のみに依存し、他の文書から参照されない）
- **ハブノード**: #3（#1から受け取り、#4と#5に分岐させる中継点）
